# **如何将模型上下文长度从4k扩展到32k？**
主要有三种思路。

第一种是不微调或少微调，只改位置编码。比如位置插值 PI，就是把训练时的位置索引线性压缩到目标长度范围内。还有 NTK‑aware 缩放，高频维度少缩放、低频维度多缩放，保留高频细节。像 YaRN 这个方法结合了 PI 和 NTK，不用微调就能把 LLaMA 2 从 4k 扩展到 128k。

第二种是微调。收集一些长文本数据，比如 Book3 或者 LongAlpaca，然后用继续预训练或者 SFT 的方式训练几百步，同时配合 LongLoRA，它使用稀疏注意力来降低计算成本，再用 LoRA 微调，这样成本很低就能扩展长度。

第三种是改注意力机制本身，比如用滑动窗口注意力加全局 token，把 O(n²) 降到 O(n × 窗口大小)。

# **评测一个“懂中文的”大模型应该测哪些维度？**
评测一个懂中文的模型，不能只看一个榜单。我会分成几个维度：

第一是知识理解，可以用 C‑Eval 或者 MMLU 的中文版。  
第二是推理能力，比如 CHID 这个成语完形填空，或者 C3 这种中文阅读理解。  
第三是代码生成，可以用 HumanEval‑X，它支持多语言并且有中文注释的版本。  
第四是长文本处理，LongBench 和 L‑Eval 里都有中文子集，测试模型处理上万字的能力。  
第五是安全合规，CValues 和 Safety‑Prompts 覆盖了政治、暴力、隐私等不安全场景。  
第六是中文特有的文化和语言现象，比如成语理解、古诗词续写、甚至方言对话。  
最后还需要开放对话的评测，比如在 Chatbot Arena 上做人类偏好比较，或者做红队测试和线上 A/B 测试。因为单一榜单很容易被刷过拟合，多个维度综合起来才更可靠。

# 如何解决大模型推理中的“重复生成”问题？
大模型在生成长文本时经常会出现重复的短语或句子，甚至陷入死循环。解决方法有几种：

第一种是解码策略层面的，比如使用重复惩罚（repetition penalty），在每一步计算概率时，对已经出现过的 token 的概率除以一个大于1的系数，让模型不那么倾向于重复。典型实现是 GPT 系列里的 `repetition_penalty=1.1` 左右。

第二种是更精细的，比如对比搜索（contrastive search），它同时考虑模型预测的概率和与之前生成内容的相似度，惩罚那些和上下文太像的 token。

第三种是训练层面的，可以在微调时增加一些抗重复的数据，或者用 unlikelihood training，明确告诉模型哪些 token 不应该再次出现。

另外，如果是长文本生成，可以结合外部记忆或检索引擎，打断模型自循环。

# 如何评估大模型的长文本能力？
长文本能力主要看三个方面：信息定位、长距离依赖和长度泛化。

常用的评估数据集有：LongBench、L-Eval、ZeroSCROLLS，它们包含长文档的摘要、多跳问答、代码补全等任务。还有一个经典的是“大海捞针”测试，就是在一篇很长的文档里随机插入一个事实，然后问模型这个事实是什么，看它能不能找出来。

关键指标包括：准确率（针对问答）、ROUGE（针对摘要）、以及是否出现中间丢失现象——就是模型能记住开头和结尾，但忘了中间的内容。另外，也会测不同长度下的性能衰减曲线，看模型能稳定处理的最大长度是多少。

长文本评测的一个难点是真实的长标签数据不好标，所以很多测试是用合成数据或者自动构造的。

#  **你会选择哪些基准评测来评估一个通用大模型的能力？请分别说明它们测的是哪个维度（推理、知识、代码、多轮对话等）。**

评估一个通用模型，我会从几个维度看：

- **知识与推理**：用 MMLU，考模型各学科知识；GSM8K 考数学推理；HellaSwag 考常识。
    
- **代码**：用 HumanEval 或 MBPP，看模型能不能写对代码函数。
    
- **多轮对话**：用 MT-Bench，看模型在多轮里能不能跟得上、答得好。
    
- **中文**：用 C-Eval 或 CMMLU，专门测中文理解。
    
- **长文本**：用 LongBench 或大海捞针测试，看模型能不能记住很长的上下文。
    
- **安全与幻觉**：用 TruthfulQA 测是不是瞎编，用 CValues 测安全合规。
    

每个维度挑一两个代表性基准就行，不用全跑。

# **最近有哪些长文本技术（如 NTK-aware scaling、YaRN、位置插值）？它们的原理是什么？在不重新训练的情况下如何扩展现有模型的上下文窗口？**

**位置插值**：最简单的方法。训练时上下文是 4k，想扩到 8k，就把所有位置索引压缩一半，原来第 8000 的位置变成 4000。这样模型看到的相对位置还在训练范围内。缺点是高频细节会模糊。

**NTK‑aware scaling**：它发现 RoPE 的不同维度对频率敏感。高频维度（短距离关系）少缩放，低频维度（长距离关系）多缩放。这样既保留局部细节，又能扩展到长距离。效果比纯插值好。

**YaRN**：目前最常用的方法。它结合了 NTK 缩放和窗口温度。具体说，它除了做 NTK 缩放，还加了一个注意力温度系数，对不同距离的 token 做不同的软化。实现简单，不微调就能把 LLaMA 从 4k 扩到 128k，困惑度几乎不涨。

**如何不重新训练扩展**：直接用这些方法修改 RoPE 的计算方式，不需要训练。比如在 HuggingFace 里改模型的 config，设置 `rope_scaling` 参数为 `{"type": "yarn", "factor": 2.0}` 就行。推理时模型就自动支持更长上下文。如果效果不够好，可以再加少量微调（几百步），但不是必须的。

#  **如果要设计一个能高效处理 1M token 上下文的模型，你会修改 Transformer 的哪些部分？**

设计一个能处理 1M token 的模型，核心就是要解决 O(n²) 的注意力计算和 O(n) 的 KV Cache 显存。我会改这几个地方：

第一，**注意力机制换成线性复杂度**。比如用滑动窗口加全局 token，或者用线性注意力（让 QK^T 变成 Q(K^T V) 的顺序），或者直接用 Mamba 这种状态空间模型，彻底不要 attention。

第二，**位置编码改成可外推的**。用 RoPE 配合 YaRN 或 NTK 缩放，确保 1M 长度下位置不崩。

第三，**KV Cache 做激进压缩**。用 GQA 或者 MQA，再加上 KV 量化（比如 INT4），甚至用 PagedAttention 分页管理。也可以引入“滑动窗口 KV”，只保留最近几万 token 的 KV，更早的丢弃或聚合。

第四，**采用稀疏注意力模式**。比如每个 token 只看到前 8k 个 token，外加每隔 1k 取一个全局 token 做长程连接。这样复杂度降到 O(n × 窗口)。

第五，**分块处理**。把 1M 切成 2k 的块，块内做全 attention，块间做少量交互。或者用 Hierarchical Transformer，先压缩成摘要再处理。

工程上，还需要模型并行（比如序列并行）和 FlashAttention 这类优化。但最根本的是：别用标准 full attention，必须改成稀疏或线性版本。

# 开放题：你在过去的项目中遇到过最棘手的大模型问题是什么？你是如何定位和解决的？**

好的，我说一个实际碰到的例子。

之前微调一个7B模型做客服对话，微调后模型确实会回答客服问题了，但通用能力崩了——问它“中国的首都是哪里”，它答非所问，甚至开始说客服话术。典型的灾难性遗忘。

**定位过程**：我先怀疑是学习率太大，调小后试了试，没改善。然后看训练数据，发现客服数据全是特定领域对话，通用问答一条没有。我算了一下数据配比，客服数据占了90%以上，通用数据太少，模型被“洗脑”了。

**解决方案**：第一，重新配比数据，把通用对话数据混进去，通用:客服从1:9调到3:7。第二，用LoRA而不是全参数微调，只更新低秩矩阵，冻结主干，保留通用知识。第三，加一个正则项，在loss里加上和原始SFT模型的KL散度惩罚。

结果：重新微调后，客服能力没掉，通用能力恢复了大半。最后总结：微调时一定要混通用数据，且用LoRA比全参数安全得多。
