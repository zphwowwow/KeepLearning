# **如何评估预训练数据的质量？数据配比怎么定？**
评估数据质量，我们通常看几个指标：一是用一个小型的高质量语言模型算困惑度，数值越低通常数据质量越高。二是去重，一般用 MinHash 算文档之间的 Jaccard 相似度，把几乎重复的文档去掉。三是做安全检测，比如用 Perspective API 看有没有毒性内容，或者正则匹配筛掉隐私信息。

数据配比没有标准答案，常用做法是先做小规模消融实验，用 1% 的数据训练几个小模型，看不同领域比例在下游任务上的表现，然后选最好的比例。还有一种方法是 DeepSeek‑V2 里提到的，监测每个领域在训练过程中的 loss 下降速度，下降慢的就提高采样权重。另外，在训练后期可以增加高质量数据的比例，这个叫数据退火，能提升最终模型的效果。

# **遇到灾难性遗忘（Catastrophic Forgetting）怎么办？**
灾难性遗忘就是在下游任务上微调后，模型忘了之前学到的通用知识。解决办法有很多种：

一种是 EWC，就是对那些重要的参数加更大的 L2 惩罚，重要程度可以用 Fisher 信息矩阵来衡量。另一种是知识蒸馏，让新模型的输出尽量接近旧模型的输出，这样能保留旧知识。还有数据回放，就是在微调的新数据里混入一部分旧数据，比如通用语料。更简单的做法是用 LoRA 这种参数高效微调，只更新少量参数，冻结主干模型，这样基本不会遗忘。

在指令微调里，常用的技巧是指令回放：每个 batch 里混入 10% 到 20% 的通用指令数据，比如 ShareGPT 里的对话，这样模型既学会了新任务，又不会忘掉怎么正常聊天。


# 假设你要从头训练一个 7B 参数的模型，如何估计需要的训练数据量？数据配比（代码、书籍、网页、多语种）一般如何确定？

**关于训练数据量**：  
根据 DeepMind 的 Chinchilla 定律，最优训练数据量大概是模型参数量的 20 倍。所以 7B 的模型，大概需要 1400 亿个 token。如果数据太少，模型学不够；数据太多，边际收益递减，训练效率就不划算了。

**关于数据配比**：  
配比没有标准答案，通常参考 LLaMA 的做法：网页数据占最大头，大概 80% 出头；代码数据 6-7%，用来提升推理和编程能力；书籍和百科数据各 4-5%，提供高质量知识。如果你的模型偏代码，比如做代码模型，代码比例可以提到 80% 以上。另外，现在很多模型还讲究课程学习，先用通用数据训练，再用专业数据精调。多语言方面，如果做中英双语，可以从中文 30%、英文 70% 开始，像 Qwen 系列中文比例会高一些，甚至到 45% 或对半分，具体看你的目标。总之，配比要靠小规模实验去调，没有万能公式。

# **在预训练过程中，发现 loss 不下降或者出现尖峰，可能的原因有哪些？你会怎么排查和解决？**

好的，这个问题在实际训练中很常见。我来分两种情况说。

**一、Loss 不下降，卡在某个值**

可能的原因有几个：

第一，学习率不合适。太大或者太小都会让 loss 下不去。我的排查方式是：先用一个 warmup 阶段，看 loss 是否能平稳下降；或者跑一个学习率扫描，看哪个范围 loss 降得最快。

第二，数据问题。最常见的是数据预处理有 bug，比如 token 化错了，或者标签和输入没对齐。我会先拿一小批数据过一遍模型，打印输入和输出的形状，确认没有全 padding 或者全 mask 的情况。另外，数据分布太偏也可能导致模型学不动，比如全是噪声文本。

第三，模型初始化有问题。比如某些层的初始化方差太大或太小，导致梯度流不畅。可以检查一下每层输出的均值和方差，看是不是在合理范围。

第四，优化器设置。比如 Adam 的 beta2 太大或者太小，或者权重衰减过大。我会先用标准的 AdamW 参数（beta1=0.9, beta2=0.95, weight decay=0.1）跑一下，排除这个因素。

**二、Loss 出现尖峰（Spike）**

尖峰通常是数值不稳定造成的，最典型的是梯度爆炸。

排查顺序：先看 loss 尖峰时有没有出现 inf 或 nan。如果有，很可能是某个层的梯度异常大。我一般会开梯度裁剪，比如把 grad norm 裁剪到 1.0，这能直接缓解。

更深层的原因可能是：数据中出现了异常长的序列或者异常大的 token id；或者是 attention 的 softmax 在输入值很大时出现下溢，虽然现在 BF16 好很多，但 FP16 仍有可能。解决办法是改用 BF16（动态范围和 FP32 一样，更稳定），或者调整 loss scaling。

另外，Pre-LN 比 Post-LN 更稳定，如果用的是 Post-LN，可以考虑换成 Pre-LN。还有，学习率 warmup 不够也会导致初期尖峰，一般 warmup 步数占总步数的 1% 左右。

**总结一下我的排查思路**：先确认是不是数值问题（看 loss 值是不是 inf），然后加梯度裁剪；再看数据预处理是否正确；然后调学习率和 warmup；最后检查模型架构和优化器设置。一般做到前三步就能解决大部分问题。

# **什么是数据“污染”？如何检测训练集中是否混入了测试集或基准评估数据？如果发现污染，应该怎么处理？**

好的，这个问题在现在大模型评测里非常关键。

**什么是数据污染？**

简单说就是：训练模型的时候，不小心把用来测试的数据也喂进去了。比如你训练一个模型，评测它的能力时用了某个榜单的测试集，但那个测试集里的样本在训练阶段就被模型见过了，那评测出来的分数就是假的，模型不是靠推理能力做对的，而是靠记忆。这就叫数据污染。

**怎么检测污染？**

核心方法是做特征匹配，看训练数据里有没有和测试数据高度重合的片段。

具体做法：第一，计算训练集和测试集之间的 n-gram 重合度，比如 13-gram 完全匹配就算疑似污染。第二，用向量检索，把测试集的样本 embedding 后去训练集里找最近邻，如果相似度很高（比如余弦相似度大于 0.98），就要警惕。第三，针对选择题或问答，可以看训练数据里是否直接出现了测试题的答案或题干原句。实际中常用一个工具叫 Google 的 Exact Substring Matching，或者直接用 Bloom 过滤器快速去重。

**发现污染后怎么处理？**

分几种情况：

如果污染程度很轻，比如只有几个样本重合，可以直接把那几条数据从训练集里删掉，然后重新训一个轻量版模型做对比，看评测分数差多少。

如果污染比较严重，比如整个测试集都混进去了，那就没法用这个榜单来评价了。这时候要么换一个没被污染的评测集，要么重新收集干净的训练数据从头训练。还有一种做法是把模型在污染数据上的表现当成“上界”，再用一个完全没有污染的测试集去测真实能力，两者对比就能知道模型到底有多少是记忆、多少是推理。

在实际的模型发布中，很多团队会刻意不去爬取已知的评测数据集，或者在数据预处理阶段就主动过滤掉和评测集有重叠的样本。比如 LLM 榜单上那些高分模型，都会公开说我们做了去污处理。

总结一句话：数据污染会严重扭曲评测结果，必须通过字符串匹配或向量相似度去检测，发现后要么删数据重训，要么换评测集。
